# fe-kit 项目汇报材料

---

## 一、项目概述

### 1) 项目是什么
**`fe-kit` 是一个前端工程化脚手架与增强 CLI（命令行工具）**：面向“新建项目”和“存量项目治理”，把工程规范、开发工具配置与团队资产（rules / skills / MCP）标准化落地。

- 一句话：**用交互式问答，一键生成或改造前端项目，把“项目创建 + 规范配置 + 团队工具接入”标准化。**

### 2) 它做什么、解决什么问题
- **解决的问题**：团队每次新项目/新仓库都要重复做初始化工作（创建目录、选框架、配构建工具、配代码规范、配编辑器、接入协作与辅助工具），容易漏、容易不一致、交付慢、维护成本高。
- **提供的能力**：
  - **`init`**：从 0 创建新项目（React 或 Vue + TypeScript，可选 Vite/Webpack/Rspack）。
  - **`enhance`**：对已有项目叠加“工程增强”（质量工具、编辑器配置、团队规则、MCP 服务接入等），目标是**增量改造**而不是推倒重来。

### 3) 面向的用户是谁
- **研发同学**：快速起项目、减少手工配置、减少踩坑。
- **团队负责人/管理者**：把“工程标准”固化为工具，降低人员变动带来的质量波动。
- **新同学/跨团队协作**：开箱即用，不依赖“口口相传”搭环境。

### 4) 在业务中的角色
**它不是业务系统，而是研发基础设施/效率工具**：让业务项目更快、更一致、更容易维护，从而缩短交付周期、降低线上风险。

## 二、项目架构说明

### 一句话总体架构总结
**`fe-kit` = “交互收集决策” → “生成/合并配置与资产” → “通过适配器写入各开发工具的标准目录（Cursor / Claude Code / VS Code / …）”。**

### 分模块说明
- **命令入口**
  像“前台接待”：接收 `init` / `enhance` 指令，负责执行主流程与兜底错误。

- **交互问答**
  像“需求采集表”：询问项目名称、路径、选 React/Vue、选构建方式、要不要路由/状态管理、要装哪些质量工具、要启用哪些团队技能/工具。

- **项目识别（增强模式）**
  像“体检报告”：自动判断当前项目大概是什么类型（React/Vue/Next/Nuxt、Vite/Webpack/Rspack、有无 TypeScript、有无 git）。

- **生成器（生成/写配置）**
  像“施工队”：拷贝模板、更新依赖与启动脚本、生成 ESLint/Prettier/Stylelint/commitlint 等规范配置。

- **适配器（对接不同工具）**
  像“安装工”：把团队能力写入不同开发工具与配置位置（**写到各工具的标准目录，而不是自造目录**），并尽量保持幂等与可增量合并。

- **模板与规则资产**
  像“标准化样板间 + 团队开发手册”：可直接复制的项目骨架与规则文档，便于复用与推广。

### 整体协作流程（数据如何流转）
用户在命令行选择需求 → 工具整理成“答案/决策对象” → 生成器按决策创建/修改目标项目文件 → 适配器把团队能力写入开发工具配置 → 形成“能跑、规范一致、工具齐全”的项目。

### 编辑器配置输出（当前项目真实落盘位置）
`init` / `enhance` 之后，根据选择的开发工具，配置会写入以下**工具标准目录**（便于透明、可预期、可被工具原生识别）：

| 开发工具 | Rules | Skills | MCP |
|---------|-------|--------|-----|
| **Cursor** | `.cursor/rules/*.mdc` | —（通过 rules 生效） | `.cursor/mcp.json`（`mcpServers`） |
| **Claude Code** | `.claude/rules/*.md` | `.claude/skills/<id>/SKILL.md` | `.mcp.json`（`mcpServers`） |
| **VS Code + Copilot** | `.github/copilot-instructions.md` | —（通过 instructions 生效） | `.vscode/mcp.json`（`servers`） |
| **CodeBuddy（CN）** | `.codebuddy/rules/*.mdc` | —（通过 rules 生效） | `.mcp.json`（`mcpServers`） |
| **Trae** | `.trae/rules/*.md`（以及规则入口文件） | `.trae/rules/<id>.md` | `.trae/mcp.json`（`mcpServers`） |
| **IDEA** | `.idea/`（当前为占位集成） | —（占位） | —（占位） |

> `.fe-kit/` 目录在当前实现中用于保存 **fe-kit 内部元数据**（例如 project 识别/执行记录等），**不作为编辑器配置输出目录**。

## 三、项目使用流程

### 场景 A：从 0 创建新项目（`init`）
1. 在终端运行 `fe-kit init`
2. 按提示填写/选择：
   - 项目名、项目路径
   - React 或 Vue
   - 路由方案（按框架给默认）
   - 状态管理方案（按框架给可选项）
   - 构建工具（Vite / Webpack / Rspack）
   - 代码质量工具（默认推荐 ESLint + Prettier）
   - 要配置到哪些开发工具（**Cursor / Claude Code / VS Code / CodeBuddy / Trae / IDEA**，可多选）
   - 要启用哪些内置 Skills（当前以 **Claude Code** 形态落盘，其它工具走 rules/instructions）
   - 要启用哪些 MCP 服务（按工具写入对应 `mcp.json` / `.mcp.json`）
3. 工具自动生成项目：
   - 拷贝对应模板（若无完整模板则生成最小可运行项目）
   - 写好依赖与启动脚本
   - 写入 `.fe-kit/` 内部元数据（用于后续增强与可追溯）
   - 写入选定工具的配置到其标准目录（例如 `.cursor/`、`.claude/`、`.vscode/`、`.github/`）

**总结**：
“新同学只要回答少量关键问题，几分钟内就能拿到一个能启动、规范齐全、团队一致的 React/Vue 项目骨架，并且 IDE/AI 工具配置自动到位。”

### 场景 B：把现有项目“一键增强”（`enhance`）
1. 进入已有项目目录（确保有 `package.json`）
2. 运行 `fe-kit enhance`
3. 工具自动识别项目类型（框架、构建工具、TS、git）
4. 选择要增强的内容：
   - 要配置到哪些开发工具（Cursor/Claude Code/VS Code/CodeBuddy/Trae/IDEA 等）
   - 要安装并配置哪些质量工具（ESLint/Prettier/Stylelint/commitlint 等）
   - 要启用哪些 Skills 与 MCP
5. 工具写入配置并完成增强：
   - 更新 `.fe-kit/` 内部元数据
   - 生成/合并质量配置文件
   - 写入对应工具配置到其标准目录（例如 `.cursor/`、`.claude/`、`.vscode/`、`.github/`）

## 四、核心功能与价值（功能 + 作用 + 价值）

- **1) 项目初始化（init）**
  - 作用：快速生成标准化前端项目骨架（React/Vue + TS，多种构建方式）。
  - 价值：把“1~2 天的手工搭建”压缩到“几分钟”，并保证起点一致。

- **2) 现有项目增强（enhance）**
  - 作用：在不推倒重来的前提下，为老项目补齐规范与工具链。
  - 价值：历史项目也能快速进入“可维护、可协作”的状态，减少“项目越做越乱”的成本。

- **3) 质量与规范工具一键配置**
  - 作用：自动安装并生成配置，减少人为差异。
  - 价值：减少低级错误、减少风格争论、提升评审效率，稳定性更可控。

- **4) 工具适配层（多编辑器/多平台）**
  - 作用：把同一套团队能力落到不同工具的**标准配置目录**（Cursor/Claude Code/VS Code/…）。
  - 价值：不被单一工具绑定，同时保证“换工具不换标准”，并且配置更透明、可审计、可回滚。

- **5) 团队规则与能力资产沉淀（rules/skills/mcp）**
  - 作用：把最佳实践做成可分发、可复用的资产。
  - 价值：新项目/新成员直接继承经验，减少口头传承与重复踩坑。

## 五、项目优势总结（把技术优势翻译成管理价值）
- **效率提升**：把重复劳动变成自动化，缩短启动与交付周期。
- **一致性提升**：多项目共享“标准答案”，减少维护成本与沟通成本。
- **可复制性强**：模板 + 规则 + 工具配置资产化，适合规模化推广。
- **对新人友好**：把隐性经验变成问答与自动生成，降低上手门槛。
- **演进空间大**：流程分层清晰，后续扩展新框架/新工具成本低。

## 七、风险与后续建议

### 风险点（影响推广/维护/稳定性）
- **增强影响边界可解释性不足**：`enhance` 会写入/合并多类配置文件（质量工具、规则、MCP、IDE 配置）。如果缺少标准化“变更清单与回滚指引”，推广时会天然担心“会不会把我项目搞坏”（这是合理顾虑）。
- **幂等与兼容性风险（Never break userspace）**：对存量项目反复执行增强，必须确保合并逻辑不丢字段、不重复污染、尽量不破坏既有脚本/配置（尤其是 `package.json` 与各类配置文件的 merge）。
- **组合爆炸的预期风险**：框架 × 构建工具 × 质量套件 × IDE 适配矩阵会带来“组合覆盖”压力；如果没有分级策略与明确支持范围，会出现预期落差。
- **依赖/引擎策略风险**：Node/包管理器要求与依赖版本策略需要明确（例如 Node ≥ 18、pnpm 版本约束），否则在存量环境会产生落地阻力。

### 建议的下一阶段规划
- **建议 1：把“可解释性”做成默认能力（变更报告 + 回滚指引）**
  每次 `enhance` 后输出一份变更摘要（新增/修改的文件、依赖变更、写入了哪些 IDE 目录、如何手动回滚关键变更）。
  管理价值：提升信任，降低推广阻力，减少沟通成本。

- **建议 2：明确支持范围与分级策略（减少组合爆炸）**
  把能力分层：必须支持的“标准套餐”（例如 React+Vite+ESLint+Prettier + Cursor/VS Code）与“实验组合”明确标识，在交互中直接提示成熟度。
  管理价值：减少预期错配与返工，确保试点可控。

- **建议 3：为增强流程补齐“回归保障”**
  最优先覆盖：配置合并不丢字段、重复执行幂等、`package.json` scripts/依赖不被破坏、IDE 配置写入可预测。
  管理价值：把“Never break userspace”从口号落到机制，降低推广的心理成本与线上风险。

- **建议 4：沉淀“公司级标准套餐”并做到一键选择**
  把最常用的组合固化成 presets（框架/构建/质量/IDE/规则/工具），减少问答项与决策成本。
  管理价值：进一步降低选择成本，提升落地速度与一致性。

- **建议 5：用可量化指标讲 ROI**
  用“启动耗时、配置缺陷数、PR 规范问题数、入职上手时间”等指标做前后对比，形成可对外扩散的试点报告模板。
  管理价值：更容易争取资源与跨团队共识。

## 八、汇总
“我们这次做的 `fe-kit`，本质上是一个前端项目的标准化‘装修队’。过去一个新项目从创建到能稳定开发，需要团队成员手工做很多配置：选框架、配构建工具、装代码规范、配编辑器/AI 工具、接入协作能力——这些事情重复、容易漏，而且每个项目做法不一样，后续维护成本很高。

`fe-kit` 把这些步骤变成了一套交互式问答：你回答项目要用 React 还是 Vue、用哪种构建方式、需要哪些质量工具、用哪些开发工具（Cursor/Claude Code/VS Code 等），它就会自动把项目骨架生成出来，或者对现有项目做增量增强，把规范与工具链补齐，并写入各工具的标准配置目录（例如 `.cursor/`、`.claude/`、`.vscode/`、`.github/`）。这样新项目可以在几分钟内进入可开发状态，而且不同项目的标准一致，新人上手也更快。

从管理角度，它的价值不在于炫技术，而在于把工程规范固化成工具：减少重复劳动、缩短交付周期、降低低级错误、提升协作效率。当前版本属于早期可用阶段，架构已经成型，下一步我们会重点补齐‘增强可解释性（变更报告/回滚指引）’与‘回归保障（幂等/兼容）’，把试点推广的风险压到最低，确保推广时不破坏现有项目的稳定性（Never break userspace）。”
